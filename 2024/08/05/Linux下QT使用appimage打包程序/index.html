2. 安装patchelf

patchelf 是一个用于修改 ELF（Executable and Linkable Format）文件的工具。ELF 文件是 Linux 和 Unix 系统上常见的可执行文件、共享库和目标文件格式。patchelf 可以用于调整和修改这些文件的某些属性，以适应特定的需求。常见的用途包括更改动态链接库的路径、修改 RPATH 和 RUNPATH 以及改变可执行文件的解释器路径。

linuxdeployqt和patchelf的关系

linuxdeployqt 在打包过程中需要处理可执行文件和共享库的依赖关系，以确保打包后的应用程序能够正确找到并加载所需的库。为此，linuxdeployqt 会使用 patchelf 来修改ELF文件（可执行文件和共享库）的某些属性，包括但不限于以下内容：

  1. 设置 RPATH/RUNPATH：

    • linuxdeployqt 使用 patchelf 来设置可执行文件和共享库的RPATH或RUNPATH，使其能够在运行时正确地找到所需的依赖库。

  2. 更改解释器路径：

    • 在某些情况下，linuxdeployqt 可能需要修改可执行文件的解释器路径，以确保应用程序在目标系统上能够正确运行。

  3. 调整库路径：

    • 当应用程序依赖的库被打包到AppDir目录中时，linuxdeployqt 使用 patchelf 修改这些库的路径，使得它们在运行时能够相互找到。

linuxdeployqt 在打包 Qt 应用程序时确实会使用到 patchelf 来处理 ELF 文件的依赖关系和路径设置。通过这种方式，linuxdeployqt 能够确保生成的 AppImage 文件包含所有必要的依赖，并且在目标系统上能够正确运行。这样，无论用户的系统环境如何，应用程序都能顺利启动并运行。

LIB_PATH和LD_LIBRARY_PATH区别

  • LIB_PATH
    LIB_PATH通常不是一个标准的环境变量，但在某些特定的构建系统或脚本中可能会被用作一个自定义变量来指定库路径。在这些情况下，它的用途取决于具体的构建工具或脚本是如何解释它的。例如，在某些自定义构建系统中，可能会使用LIB_PATH来指定链接时查找库的路径。

  • LD_LIBRARY_PATH
    LD_LIBRARY_PATH是Unix和Linux系统上的一个标准环境变量。它用于指定在运行时动态链接器查找共享库的目录。当一个程序运行时，动态链接器会按照以下顺序查找共享库：

    1. 程序的运行时路径（通常由编译器设置）。
    2. 环境变量LD_LIBRARY_PATH指定的目录。
    3. 系统默认的共享库路径（通常是/lib和/usr/lib等目录）。

    在运行时，如果LD_LIBRARY_PATH指定了目录，动态链接器会优先在这些目录中查找共享库。

重要提示：

  • 因为linuxdeployqt将在内部使用ldd命令，来确定从何处将库复制到包中。ldd命令本身只能查找隐式加载的so，故若程序中代码使用显式加载的so，它是找不到的，这时需要开发者自己去保证显式加载so的路径可被搜索。
  • 举个例子：若存在程序A显式加载B.so，B.so隐式加载C.so，则对程序A使用linuxdeployqt拷贝依赖时，程序A的间接依赖文件C.so，将不会被拷贝。当程序跑起来加载B.so时，就会报找不到C.so的错误。这一点尤其需要注意。此时我们需要再对B.so使用一次linuxdeployqt拷贝依赖。这样程序A加载B.so，B.so加载C.so，就都可以找到了。

--appimage-extract-and-run 是一个用于AppImage文件的命令行选项。AppImage是一种便携式软件打包格式，主要用于在Linux系统上分发应用程序。它允许开发者将应用程序及其所有依赖项打包成一个单一的可执行文件，用户可以通过下载并运行这个文件来使用应用程序，而无需进行传统的安装过程。

当你使用 --appimage-extract-and-run 选项运行AppImage文件时，会发生以下情况：

  1. 提取内容：AppImage文件的内容会被提取到一个临时目录中。
  2. 运行应用程序：从提取的临时目录中直接运行应用程序。

这种方式的优点在于，可以在不实际提取文件到系统的情况下运行应用程序，同时也方便进行调试和测试。

  • 示例

假设你有一个名为 example.AppImage 的AppImage文件，你可以使用以下命令来提取并运行它：

  • 工作原理

  1. 提取内容：

    • AppImage文件本质上是一个压缩文件，包含了应用程序及其所有依赖项。
    • 使用 --appimage-extract-and-run 选项时，AppImage会在一个临时目录（通常是 /tmp 目录下）中解压缩其内容。

  2. 运行应用程序：

    • 在提取内容之后，AppImage会找到主可执行文件并运行它。
    • 运行完成后，临时目录会被清理（如果程序正常退出）。

  • 使用场景

    • 调试：在开发和测试过程中，可以使用 --appimage-extract-and-run 选项来查看AppImage内部的文件结构，并直接运行应用程序，以便快速找到并解决问题。
    • 临时运行：如果你只想临时运行一个AppImage文件，而不想提取其内容到系统的某个固定位置，这个选项非常方便。
    • 环境隔离：有时你可能希望在一个临时隔离的环境中运行应用程序，以确保它不会影响系统的其他部分。

  • 注意事项

    • 由于内容被提取到临时目录中，因此在运行过程中可能会占用一些额外的磁盘空间。
    • 这个选项主要用于调试和测试，通常情况下，直接运行AppImage文件（不使用 --appimage-extract-and-run 选项）会更加高效。
